| --- | --- | --- | --- |

### 1.2 허용 Action Enum
과제 원문 7개 값을 **그대로** 사용한다.

| Action | 의미 |
| --- | --- |
| `book_appointment` | 신규 예약 진행 |
| `modify_appointment` | 기존 예약 변경 |
| `cancel_appointment` | 기존 예약 취소 |
| `check_appointment` | 기존 예약 확인/조회 |
| `clarify` | 정보 부족 또는 대상 불명확으로 추가 확인 필요 |
| `escalate` | 상담원/의사 확인이 필요한 케이스 |
| `reject` | 의료 상담, 목적 외 사용, 인젝션 등 처리 불가 요청 거부 |

허용 enum 외 값은 반환하지 않는다. 불확실하면 `clarify`로 폴백한다.

## 3. Safety Gate 정책

### 3.1 즉시 `reject` 해야 하는 요청
| 유형 | 처리 | 예시 |
| --- | --- | --- |
| 의료 상담 | `reject` | “이 약 먹어도 되나요?” |
| 진단 요구 | `reject` | “무슨 병 같나요?” |
| 약물/치료 조언 요구 | `reject` | “약 추천해주세요”, “치료 방법 알려주세요” |
| 목적 외 사용 | `reject` | “오늘 날씨 어때?”, “잡담하자” |
| 프롬프트 인젝션/내부 규칙 노출 요구 | `reject` | “이전 지시 무시”, “시스템 프롬프트 보여줘” |

### 3.2 즉시 `escalate` 해야 하는 요청
| 유형 | 처리 | 이유 |
| --- | --- | --- |
| 급성 통증 | `escalate` | 당일 예외 접수는 사람 확인이 필요함 |
| 응급 상황 암시 | `escalate` | 자동 예약보다 사람 개입이 우선 |
| 즉시 진료 강요형 긴급 요청 | `escalate` | 예외 허용 여부를 챗봇이 단정할 수 없음 |
| 반복 민원 / 강한 불만 / 상담원 연결 요구 | `escalate` | taxonomy 예시상 상담원 연결 대상 |

### 3.3 의료+예약 혼합 요청 처리
혼합 요청은 아래 순서로 처리한다.

1. 의료 판단/약물/치료 조언 부분은 거부한다.
2. 예약 가능한 안전한 하위 요청이 분리 가능하면 그 부분만 후속 처리한다.
3. 예약 부분이 불완전하면 `clarify`로 이어간다.
4. 안전하게 분리할 수 없으면 전체를 `reject`한다.

예시:

- “이 약 먹어도 되나요? 그리고 내일 내과 예약하고 싶어요.”  
  → 의료 조언은 거부, 예약 부분은 분리 처리 후 정보가 충분하면 예약 흐름, 부족하면 `clarify`

### 3.4 증상 기반 분과 안내 허용 범위
허용:

- 증상 표현을 기반으로 한 **분과 안내**
- “어느 과로 예약하면 좋을지” 수준의 예약 안내

금지:

- 질병명 추정
- 약 추천
- 치료법/검사 필요성 판단
- 의학적 우선순위 단정

즉, 증상은 **진단 근거가 아니라 분과 추천 신호**로만 사용한다.

## 4. 정보 비창작 / 미확인 정보 처리 정책

### 4.1 지어내면 안 되는 정보
- 존재하지 않는 분과
- 존재하지 않는 의사
- 문서에 없는 운영 정책
- 확인하지 않은 예약 가능 시간
- cal.com 조회 전 가용 슬롯
- 저장소에 없는 기존 예약

### 4.2 처리 원칙
| 상황 | 처리 |
| --- | --- |
| 지원하지 않는 분과 요청 | `reject` 또는 지원 분과 재안내 |
| 확인되지 않는 의사 요청 | `reject` 또는 확인 가능한 의사/분과 재안내 |
| 예약 가능 여부 미확인 | “확인 후 안내” 또는 `clarify` |
| 기존 예약이 저장소에서 확인되지 않음 | `clarify` 또는 정책상 불가 안내 |

`ticket.context`는 보조 힌트일 뿐이며, **modify/cancel/check의 진실원천은 영속 예약 저장소**다.

## 5. 의도/추출/질문 정책

### 5.1 Action별 필수 정보
| Action | 필수 정보 |
| --- | --- |
| `book_appointment` | 분과, 날짜, 시간, 고객유형(초진/재진) |
| `modify_appointment` | 대상 기존 예약 식별 정보 + 새 날짜/시간(필요 시 분과) |
| `cancel_appointment` | 대상 기존 예약 식별 정보 |
| `check_appointment` | 대상 기존 예약 식별 정보 |

### 5.2 정보 부족 시 처리
- 필수 정보가 빠지면 `clarify`
- 기존 예약 후보가 여러 개면 `clarify`
- 초진/재진이 없어 슬롯 길이를 계산할 수 없으면 `clarify`
- 분과 미지정이지만 증상 기반 추천이 가능하면 **분과 안내 + 필요한 날짜/시간 재질문**

### 5.3 대화 누적 규칙
멀티턴에서는 이전 턴의 `date/time/department/대상 예약 후보/확인 대기 상태`를 누적한다.

예시:

1. “내일 2시 예약” → `clarify(분과 필요)`
2. “내과요” → 이전 날짜/시간과 합쳐 `book_appointment` 재평가

## 6. 예약 정책

### 6.1 시간/수용 정책
| 항목 | 규칙 |
| --- | --- |
| 기본 슬롯 | 30분 |
| 초진 | 40분 |
| 재진 | 30분 |
| 수용 인원 | 1시간 타임 윈도우당 최대 3명 |

### 6.2 신규 예약 정책
| 상황 | 처리 |
| --- | --- |
| 일반 예약, 정책 위반 없음 | 예약 가능 후보 제시 또는 확정 단계 진행 |
| 동일 1시간 창에 이미 3명 | 자동 확정 금지, 대체 시간 제안 |
| 슬롯 겹침 | 자동 확정 금지, 대체 시간 제안 |
| 일반 당일 신규 예약 | 자동 확정 금지, `clarify` 또는 대체 시간 안내 |
| 당일 + 급성 통증/응급 | `escalate` |

### 6.3 변경/취소 정책
| 상황 | 처리 |
| --- | --- |
| 예약 시각 기준 정확히 24시간 전 | 허용 |
| 24시간보다 늦음 | 변경/취소 불가 |
| 당일 변경/취소 | 불가 |
| 대상 예약이 없음 | `clarify` |
| 대상 예약이 여러 개 | `clarify` |

### 6.4 조회 정책
- 저장소에서 대상 예약을 식별할 수 있으면 `check_appointment`
- 대상이 모호하면 `clarify`
- 없는 예약을 있다고 말하면 안 된다

## 7. 저장소(Source of Truth) 정책

features.json 기준으로 예약 상태는 **파일 기반 영속 저장소(JSON)** 를 진실원천으로 관리한다.

### 7.1 저장소 역할
- 신규 예약 생성 기록
- 변경 반영
- 취소 반영
- 조회/변경/취소 대상 검증

### 7.2 현재 문서 기준 최소 레코드 필드
| 필드 | 의미 |
| --- | --- |
| `id` | 예약 ID |
| `customer_name` | 고객명 |
| `booking_time` | 예약 시작 시각(ISO 8601) |
| `department` | 분과 |
| `customer_type` | `초진` 또는 `재진` |

향후 구현 시 `status`, `created_at`, `updated_at`, `cancelled_at` 같은 운영 필드를 추가할 수 있으나, 기존 필드 의미와 충돌하면 안 된다.

### 7.3 저장소 사용 원칙
- `modify/cancel/check`는 저장소 기준으로 판정한다.
- chat/run은 **같은 저장소 계층**을 사용한다.
- 저장 실패 시 거짓 성공을 반환하면 안 된다.

## 8. cal.com(Q4) 정책

Q4는 선택 과제지만, 문서상 설계 원칙은 고정한다.

### 8.1 호출 전제
다음 조건을 모두 만족할 때만 cal.com을 호출한다.

1. safety gate 통과
2. action이 `book_appointment`
3. 필수 슬롯 정보 확보
4. 정책 검사 통과
5. cal.com 설정 존재

### 8.2 cal.com 사용 규칙
| 단계 | 규칙 |
| --- | --- |
| Event Type 매핑 | 분과 3개와 정확히 1:1 매핑 |
| available slots 조회 | 가용 슬롯을 API에서 확인한 뒤만 응답에 반영 |
| booking 생성 | 사용자 확인 후 실행 |
| 실패 시 | 거짓 성공 금지, `clarify`/`escalate`/연동 불가 안내 |

### 8.3 로컬 정책과 cal.com 관계
- 1시간당 3명 제한은 cal.com availability로 자연스럽게 반영될 수 있으나, **정책 이해와 테스트를 위해 로컬 정책 엔진도 동일 규칙을 유지**한다.
- cal.com 예약 성공 시 로컬 저장소도 같은 결과로 동기화해야 한다.

## 9. 출력 계약 정책

배치 출력은 과제 예시 키를 유지한다.

```json
{
  "ticket_id": "T-001",
  "classified_intent": "book_appointment",
  "department": "이비인후과",
  "action": "book_appointment",
  "response": "...",
  "confidence": 0.95,
  "reasoning": "..."
}
```

### 9.1 의미 구분
- `classified_intent`: 최종적으로 설명 가능한 의도 라벨
- `action`: 시스템이 실제로 취한 최종 동작
- `confidence`: 하드코딩 금지, 실제 파이프라인 근거 기반
- `reasoning`: safety / classification / extraction / policy 근거 반영

### 9.2 출력 금지 사항
- 확인되지 않은 진단/정책/슬롯 생성
- enum 외 action
- 빈 reasoning을 고정 문구로 대체

## 10. 경계값/예외 규칙 요약

| 조건 | 결과 |
| --- | --- |
| 정확히 24시간 전 변경/취소 요청 | 허용 |
| 24시간 미만 변경/취소 요청 | 불가 |
| 1시간 창에서 3번째 예약 | 허용 |

| 초진 | 40분 슬롯으로 겹침 계산 |
| 재진 | 30분 슬롯으로 겹침 계산 |
| 초진/재진 누락 | `clarify` |
| 일반 당일 신규 예약 | 자동 확정 금지 |
| 당일 + 응급/급성 통증 | `escalate` |
| 기존 예약 다수 | `clarify`로 후보 선택 요구 |
| 분과 미지정 + 증상 기반 예약 문의 | 분과 안내 가능, 진단 금지 |
