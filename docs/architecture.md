## Q2 Agent 아키텍처

### 1. 문서 목적

본 문서는 코비메디 예약 Agent PoC의 Q2 아키텍처 골격을 설명한다. 설계 우선순위는 `safety > correctness > policy compliance > demo polish`이며, 과제 요구사항과 `AGENTS.md`의 비타협 조건을 그대로 따른다.

핵심 설계 원칙은 다음과 같다.

- `chat.py`와 `run.py`는 동일한 `src/agent.py` 로직을 공유한다.
- 안전성 게이트가 모든 일반 처리보다 먼저 실행된다.
- 정책 규칙은 `policy.py`에서 결정론적으로 우선 판정한다.
- LLM은 자유 생성기가 아니라 **의도 해석과 자연어 응답 보조기**로 제한한다.
- Action 값은 과제 원문의 7개 enum만 사용한다.

### 2. 전체 아키텍처 개요

#### 2.1 End-to-End 텍스트 다이어그램

```text
[입력]
  ├─ 인터랙티브 입력(chat.py)
  └─ 배치 티켓(run.py)
          ↓
[공통 엔트리포인트: src/agent.py]
          ↓
[안전성 게이트]
  ├─ 의료 상담 요청 차단 -> reject
  ├─ 목적 외 사용 차단 -> reject
  ├─ 프롬프트 인젝션 차단 -> reject
  └─ 급성 통증/응급 징후 감지 -> escalate
          ↓
[의도 분류]
  ├─ 예약/변경/취소/조회/clarify 후보 판별
  ├─ 분과 추정
  └─ 누락 정보 식별
          ↓
[정책 검사]
  ├─ 24시간 변경/취소 제한
  ├─ 1시간당 최대 3명 제한
  ├─ 초진 40분 / 재진 30분 규칙
  ├─ 기존 예약 존재 여부 검증
  └─ 자동 처리 가능 여부 결정
          ↓
[응답 생성]
  ├─ policy 결과 기반 결정론 응답
  ├─ 필요 시 LLM으로 자연어 표현 정리
  └─ 과제 JSON 스키마로 구조화
          ↓
[출력]
  ├─ chat.py: 사용자 대화 응답
  └─ run.py: results.json
```

#### 2.2 컴포넌트 역할 요약

| 단계 | 책임 | 실패 시 기본 동작 |
| --- | --- | --- |
| 입력 | 메시지/티켓 수집, 공통 포맷 정규화 | 필수 필드 부족 시 `clarify` |
| 안전성 게이트 | 의료 상담, 목적 외 사용, 인젝션, 응급성 차단 | `reject` 또는 `escalate` |
| 의도 분류 | 예약 관련 action 후보와 분과 추정 | 불명확 시 `clarify` |
| 정책 검사 | 병원 규칙을 결정론적으로 적용 | 위반 시 정책 안내 또는 대체 제안 |
| 응답 생성 | 사용자 친화적 문장 + JSON 계약 충족 | 보수적으로 짧고 안전한 응답 |

### 3. chat/run 공통 로직 구조 — 왜 `agent.py`를 공유하는가

#### 3.1 제안 구조

```text
chat.py
  └─ 사용자 입력을 받아 agent.handle_interactive(...) 호출

run.py
  └─ 티켓 JSON을 순회하며 agent.handle_ticket(...) 호출

src/agent.py
  ├─ 입력 정규화
  ├─ safety gate 실행
  ├─ classifier 호출
  ├─ policy 판정
  ├─ response builder 호출
  └─ 공통 결과 객체 반환
```

#### 3.2 공유해야 하는 이유

1. **판정 일관성 확보**  
   인터랙티브 데모와 배치 평가가 서로 다른 로직을 사용하면, 같은 문장에 대해 다른 action이 나오는 문제가 생긴다. Hidden Test에서는 이런 불일치가 바로 드러난다.

2. **안전 규칙의 단일 진실원(Single Source of Truth)**  
   의료 상담 차단, 인젝션 거부, 응급 에스컬레이션 규칙은 한 곳에서만 관리해야 누락이 없다. `agent.py` 공유 구조는 안전 정책 누수를 줄인다.

3. **테스트 가능성 향상**  
   테스트는 `src/agent.py`의 핵심 함수만 검증하면 chat/run 모두를 동시에 커버할 수 있다. 이는 회귀 테스트와 Hidden Test 대응에 유리하다.

4. **유지보수 비용 절감**  
   정책 변경이 발생했을 때 chat/run을 각각 수정하면 드리프트가 생긴다. 공통 로직 구조는 변경 비용과 오류 가능성을 함께 낮춘다.

### 4. safety-first 파이프라인 — 왜 안전성을 최우선 배치했는가

#### 4.1 안전성 우선 배치 이유

코비메디 과제의 가장 강한 제약은 **의료 상담 금지**, **목적 외 사용 거부**, **허위 정보 금지**다. 따라서 예약 성공률보다 먼저, “하면 안 되는 응답을 절대 하지 않는 구조”가 필요하다. 특히 Q3 요구사항은 의료 질문 오응답을 사실상 0건에 가깝게 줄이는 방향을 요구하므로, 안전성은 후처리 옵션이 아니라 **파이프라인의 첫 단계**여야 한다.

안전성 게이트를 앞단에 두는 이유는 다음과 같다.

- LLM이 먼저 답하게 하면 의료 질문에 대해 과도한 생성이 발생할 수 있다.
- 정책 검사보다 먼저 위험 요청을 차단해야 불필요한 해석·추론을 줄일 수 있다.
- 목적 외 사용과 인젝션은 예약 파이프라인에 들어오기 전에 차단해야 시스템 경계를 지킬 수 있다.

#### 4.2 안전성 게이트의 처리 대상

- **의료 상담 요청**: 진단, 약 복용 가능 여부, 치료 방법, 검사 해석 요청 → `reject`
- **목적 외 사용**: 잡담, 날씨, 병원 예약과 무관한 질문 → `reject`
- **프롬프트 인젝션**: 시스템 프롬프트 공개, 정책 무시 요구 → `reject`
- **응급/급성 통증**: 자동 예약 대신 사람 개입 필요 → `escalate`

#### 4.3 혼합 요청 처리 원칙

예: “감기 같은데 내일 내과 예약도 잡아주세요.”  
이 경우 Agent는 의료 판단 부분은 거부하고, 예약 처리 가능 부분만 분리해 안전하게 안내해야 한다. 즉, 안전성 게이트는 무조건 대화를 종료하는 장치가 아니라 **위험 요소를 제거한 뒤 예약 가능 범위만 남기는 필터**로 동작해야 한다.

### 5. 결정론 policy + LLM hybrid — 왜 분리했는가

#### 5.1 분리 원칙

이 Agent는 모든 판단을 LLM 하나에 맡기지 않는다. 대신 다음처럼 역할을 분리한다.

- **결정론 policy (`policy.py`)**: 병원 규칙, 경계값, 허용/불가 판정
- **LLM (`Ollama qwen3-coder:30b`)**: 자연어 해석, 누락 정보 보조 추출, 응답 문장 정리

#### 5.2 policy를 결정론으로 두는 이유

아래 규칙은 변동 없이 정확하게 재현되어야 한다.

- 당일 변경/취소 불가, 24시간 기준
- 1시간당 최대 3명
- 초진 40분 / 재진 30분
- 기존 예약 유무에 따른 modify/cancel/check 가능 여부
- 허용된 7개 action enum만 반환

이러한 규칙을 LLM 자유 추론에 맡기면 다음 문제가 생긴다.

- 같은 입력에도 출력이 흔들릴 수 있음
- 24시간 경계값, 3명 제한 같은 숫자 규칙에서 오류가 날 수 있음
- Hidden Test의 edge case에서 설명은 그럴듯하지만 action이 틀릴 수 있음

따라서 정책은 코드로 고정하고, LLM은 규칙 바깥의 언어적 모호성만 처리하도록 제한하는 편이 더 안전하다.

#### 5.3 LLM을 병행하는 이유

반대로 자연어 입력은 완전한 규칙 기반만으로 처리하기 어렵다.

- “목이 너무 아픈데 어느 과로 가야 하나요?” 같은 분과 추정형 문장
- “예약 바꾸고 싶은데 수요일 말고 목요일요” 같은 축약 표현
- 누락 정보가 많지만 예약 의도는 분명한 문장

이 영역에서 LLM은 표현 다양성을 흡수하는 데 유리하다. 즉, **정책 판단은 코드가 하고, 언어 해석은 LLM이 보조하는 구조**가 과제 요구와 일반화 성능을 동시에 만족시키기 쉽다.

### 6. Ollama 로컬 모델(`qwen3-coder:30b`) 선택 이유

#### 6.1 선택 근거

1. **로컬 실행으로 개인정보/운영 리스크 완화**  
   환자 메시지를 외부 SaaS API로 보내지 않고 로컬에서 처리할 수 있어 PoC 단계에서 보안 설명이 쉽다.

2. **재현 가능성 확보**  
   모델명과 실행 환경을 고정하면 데모와 배치 평가에서 비교적 일관된 결과를 얻기 쉽다.

3. **구조화 출력 적합성**  
   과제는 `classified_intent`, `department`, `action`, `confidence`, `reasoning` 등 JSON 스키마를 요구한다. Ollama의 `format='json'` 사용은 파싱 안정성을 높인다.

4. **한국어 지시 이해 + 규칙 준수 보조**  
   본 과제는 한국어 사용자 입력과 정책 문서를 다루므로, 한국어 문맥을 무리 없이 처리할 수 있는 모델이 적합하다.

5. **코드/구조화 친화성**  
   qwen 계열 coder 모델은 지시 따르기와 구조화 응답에서 상대적으로 유리해, 자유 대화형 생성보다 “정해진 슬롯을 채우는 작업”에 맞다.

#### 6.2 사용 원칙

- `format='json'`으로 구조화 출력 강제
- 허용된 action enum 외 값은 후단 validator에서 거부
- policy 위반 가능성이 있으면 LLM 결과보다 결정론 정책을 우선
- 모르면 추측하지 않고 `clarify` 또는 보수적 거절

### 7. Hidden Test 대응 전략

Hidden Test의 핵심은 50개 공개 예시에 대한 과적합이 아니라, **정책 이해와 일반화 가능성**을 검증하는 것이다. 따라서 대응 전략도 “케이스 암기”가 아니라 “파이프라인 안정화”에 맞춰야 한다.

#### 7.1 일반화 중심 설계

- 문장 패턴 매칭보다 **의도/정책 분리 구조**를 사용한다.
- 공개 티켓의 표현을 외우지 않고, action taxonomy와 policy digest를 기준으로 판정한다.
- 분과 추정은 증상 단어 사전 + LLM 보조 해석의 조합으로 설계한다.

#### 7.2 경계값 테스트 우선

Hidden Test에서 자주 나올 수 있는 경계 사례를 명시적으로 방어한다.

- 변경/취소 요청 시각이 정확히 24시간 전인지 여부
- 1시간 윈도우 3명/4명 경계
- 초진/재진 미확정 상태
- 기존 예약 없음 + modify/cancel/check 요청
- 의료 질문과 예약 요청이 섞인 복합 문장
- 인젝션 문구가 예약 문장처럼 위장된 경우

#### 7.3 출력 계약 엄수

Hidden Test는 의미뿐 아니라 포맷도 볼 가능성이 높다. 따라서 아래를 엄격히 유지한다.

- action은 정확히 7개 enum만 사용
- 결과 JSON 키는 과제 예시와 동일하게 유지
- 불확실할 때는 과감히 `clarify`, 위험하면 `reject`/`escalate`
- 없는 정책을 지어내지 않음

#### 7.4 회귀 방지 전략

- Safety test, policy test, batch output schema test를 분리한다.
- `chat.py`와 `run.py`가 동일한 `agent.py`를 호출하는지 테스트로 고정한다.
- LLM 출력은 validator로 감싸 비허용 값이 나오면 fallback 경로를 사용한다.

### 9. Q4 cal.com 연동 계획 (선택과제)

현재 코드베이스는 Q2/Q3 중심의 로컬 정책·로컬 데이터 기반 PoC이며, cal.com 연동은 아직 미구현 상태다. 다만 공식 문서 기준으로 Q4는 **API v2 + API Key(or OAuth)** 조합으로 구현 가능하며, 현재 확인된 최단 경로는 API key 기반 연동이다.

#### 9.1 외부 선행 준비 체크리스트

Q4 구현 전 cal.com 대시보드에서 아래 준비가 완료되어야 한다.

- cal.com 계정 생성
- API Key 발급
- Event Type 3개 생성
  - `ent-consultation` (이비인후과)
  - `internal-medicine` (내과)
  - `orthopedics` (정형외과)
- 각 Event Type이 실제 예약 가능 상태인지 확인
- 병원 데모용 캘린더 연결 및 timezone 확인
- 최종 데모 시 cal.com 캘린더 생성 스크린샷 1~2장 확보

#### 9.2 구현 대상 범위

Q4는 기존 `book_appointment` 흐름을 아래처럼 확장한다.

```text
book_appointment 판정
  ↓
분과 → cal.com Event Type 매핑
  ↓
cal.com available slots 조회
  ↓
가용 시간 제안 응답
  ↓
사용자 확인
  ↓
cal.com booking 생성
  ↓
생성 결과를 chat/run 공통 응답에 반영
```

핵심은 `chat.py`, `run.py`가 직접 cal.com을 호출하지 않고, 반드시 `src/agent.py` 공통 로직과 별도 client 계층을 통해 처리하는 것이다.

#### 9.3 제안 모듈 추가/변경

| 파일 | 변경 내용 |
| --- | --- |
| `src/agent.py` | `book_appointment`에 대해 slot 조회/booking 생성 오케스트레이션 추가 |
| `src/calcom_client.py` | cal.com API 호출 전담 (`get_available_slots`, `create_booking`) |
| `src/config.py` 또는 동등 설정 계층 | `CALCOM_API_KEY`, base URL, API version header 로드 |
| `src/response_builder.py` | slot 제안/booking 성공/외부 실패 응답 템플릿 확장 |
| `run.py`, `chat.py` | 공통 Agent 인터페이스 사용만 유지, 중복 로직 추가 금지 |

#### 9.4 정책 책임 분리

Q4에서도 다음 원칙을 유지한다.

- 안전성 게이트는 여전히 최우선이다.
- 응급/의료 상담/목적 외 사용은 cal.com 호출 전에 차단한다.
- `policy.py`는 24시간 변경/취소 제한, 응급 escalation 등 **병원 정책 판정**을 계속 담당한다.
- 실제 슬롯 가능 여부는 cal.com `available slots` 응답을 우선 신뢰한다.
- 외부 API 실패 시 절대 예약 완료라고 말하지 않는다.

#### 9.5 구현 단계 계획

1. `features.json`에 Q4 항목 추가
2. cal.com 설정값 로드 계층 추가
3. cal.com API client 작성
4. `book_appointment` 2단계 흐름 구현
5. mock 기반 테스트 추가
6. 실제 API smoke test 및 캘린더 스크린샷 확보

#### 9.6 현재 상태 기준 구현 리스크

현재는 Event Type 3개가 아직 생성되지 않았으므로, 즉시 실 booking까지 검증할 수는 없다. 따라서 구현 초기에는 다음 전략이 합리적이다.

- 먼저 문서와 feature에 Q4 범위를 고정
- 코드에서는 설정/어댑터/안전 폴백 구조부터 구현
- Event Type 생성 후 실제 slot 조회 및 booking 생성으로 마무리 검증

### 10. 결론

Q2 Agent의 핵심은 “LLM이 알아서 다 하는 예약 봇”이 아니라, **안전성 게이트와 결정론 정책을 중심에 두고 LLM을 제한적으로 사용하는 예약 자동화 파이프라인**이다. 이 구조는 과제의 Hard Constraint를 우선 충족하면서도, 인터랙티브 데모와 배치 평가를 동일한 로직으로 일관되게 처리할 수 있게 해준다.

특히 `safety-first`, `shared agent core`, `deterministic policy + LLM hybrid`, `local structured model`이라는 네 가지 축은 Hidden Test와 실제 운영 리스크를 동시에 방어하기 위한 설계 결정이다.

Q4는 이 구조를 뒤집는 것이 아니라, `book_appointment` 성공 경로 뒤에 **외부 스케줄러 연동 계층**을 붙이는 확장 작업이다. 따라서 Q4를 구현하더라도 중심 원칙은 그대로 유지되어야 하며, 외부 연동 실패 역시 safety-first 철학 아래 보수적으로 처리되어야 한다.
